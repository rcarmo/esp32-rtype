# ESP32 CYD 2432S028 target

## Role

Small ESP32 compatibility target for keeping the codebase portable to the original CYD-style SPI display board. It is not the primary performance target.

PlatformIO environment:

```text
esp32-cyd-rtype
```

Board define:

```c
RTYPE_BOARD_ESP32_2432S028
```

Build:

```bash
make build-cyd
```

## Hardware characteristics

- MCU: classic ESP32.
- Display: 240x320 ILI9341 SPI panel.
- Orientation: physical portrait panel used for a rotated landscape game view.
- Much less RAM and display bandwidth than the S3 RGB-panel target.
- SPI display transfers dominate runtime cost more than the M72 source renderer.

## Rendering strategy

The CYD strategy is a reduced-bandwidth, board-specific display path over the shared M72 renderer.

### Source and viewport

- The M72 logical source remains 384x256.
- CYD output uses a logical 320x240 landscape view mapped onto the physical 240x320 portrait display.
- The aspect-correct game viewport is 320x213, avoiding non-proportional stretch.
- The CYD code rotates the game into portrait-oriented physical column transfers.

### Strip/column rendering

- CYD cannot assume a large RGB-panel framebuffer.
- The display path renders and flushes strips/columns sized for SPI bandwidth.
- Column lookup tables precompute source X/Y mapping for the rotated physical layout.
- Background-only and composited column paths exist so the SPI worker can prioritize bandwidth and sprite regions.
- `rtype_m72_video_render_cyd_columns()` and related helpers sample the shared M72 tile/sprite state for the CYD layout.

### Asynchronous display worker

- CYD display updates run through a FreeRTOS worker pinned to core 0.
- The producer queues newest-frame work and can drop stale queued work if the SPI display falls behind.
- The goal is smooth live motion under SPI bandwidth limits, not full S3-class frame pacing.

## Fidelity rules for CYD

- CYD must still render live emulator state; no static or prebaked frames as final output.
- Board-specific shortcuts must not be copied into the S3 path unless they preserve MAME-equivalent pixels.
- Keep CYD building after shared M72 renderer changes.
- If S3-only caches/fast paths are introduced, guard them with S3 board defines so CYD remains portable.

## Limitations

- SPI display bandwidth is the main bottleneck.
- RAM is substantially tighter than on S3.
- CYD is not expected to match the S3 frame rate or exact full-frame RGB-panel strategy.
- Full live M72 fidelity is constrained by the output path.
- Audio and real input remain out of scope.

## Validation

```bash
make build-cyd
```

If hardware is attached, flash manually with the appropriate serial port:

```bash
make flash PIO_ENV=esp32-cyd-rtype SERIAL_PORT=/dev/serial/by-id/...
```

The S3 target remains the primary place to validate MAME fidelity, queue-drain behavior, and background restoration.
