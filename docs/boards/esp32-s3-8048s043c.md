# ESP32-S3 8048S043C target

## Role

Primary hardware target for the live R-Type/M72 port.

PlatformIO environment:

```text
esp32-s3-8048s043c-rtype
```

Board define:

```c
RTYPE_BOARD_ESP32_8048S043C
```

Canonical build/deploy:

```bash
make build-s3
make deploy-s3 SERIAL_PORT=/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0
```

## Hardware characteristics

- Board family: Sunton ESP32-8048S043C style ESP32-S3 board.
- MCU: ESP32-S3 with PSRAM.
- Display: 800x480 RGB panel.
- Touch: GT911 on the board, not currently part of the R-Type gameplay path.
- Primary serial adapter: CH340-style USB serial, usually `/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0`.

Display mapping:

```text
M72 source framebuffer: 384x256 RGB565
Panel view:             720x480+40,0
Panel native:           800x480
```

The game is scaled proportionally with black side bars. Do not stretch to 800x480.

## Flash layout

The S3 partition table reserves separate firmware, main CPU ROM map, and FAT ROM-storage regions.

```text
maincpu: 0x410000, 1MB
storage: 0x510000, 9MB
```

Data flash command:

```bash
make flash-s3-data SERIAL_PORT=/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0
```

This writes:

```text
0x410000 artifacts/packed-rtype/maincpu-map.bin
0x510000 artifacts/s3-rtype-fatfs-wl-9m.bin
```

Graphics ROMs are loaded from:

```text
/spiflash/rtype
```

## Emulator/runtime approach

- Main CPU fetches are flash-mapped from the `maincpu` partition.
- Work RAM, sprite RAM, palette RAM, VRAM0/1, and M72 sound/shared RAM are sparse PSRAM-backed regions.
- The firmware uses an incremental 8086/V30-family interpreter that covers the R-Type runtime path.
- The renderer consumes live M72 state: VRAM, sprite buffer, palette, scroll registers, video-off state.

## Rendering strategy

The S3 strategy is a full source-framebuffer renderer followed by an asynchronous RGB-panel handoff.

### Frame production

- The emulator renders into a 384x256 RGB565 source framebuffer.
- The app allocates three source framebuffers on S3.
- Rendering is tied to the live V30/M72 state, not to prebaked images.
- Before each snapshot render, the app gives the M72 update queue a bounded drain window.
- This queue-drain phase is required because R-Type clears and rebuilds background tile attributes during frame work. Rendering mid-update produces missing/dark backgrounds.
- Serial `S3 PERF` logs expose this as `qdrain=ok/miss`; `make smoke-s3` requires active `qdrain=<nonzero>/0`.

### MAME fidelity in the renderer

Implemented MAME-aligned behavior:

- M72 visible X range is raw `64..447`, mapped to 384 output pixels.
- Tile/sprite plane decode uses MAME bit order.
- Palette RAM mirrors disconnected A9 behavior and returns high read bits set.
- S3 maps M72 sound/shared RAM at `0xe0000..0xeffff` instead of treating it as a reset ROM mirror.
- Tile priority/group masks follow MAME `m72_v.cpp` transmask setup.
- Firmware draws all MAME tile passes, including initial layer-0 bitmap writes used during priority-buffer preparation.
- Sprite traversal follows MAME sprite-list stepping (`offs += width * 8 bytes`) and reverse draw order.
- Fully offscreen sprites are clipped before decode.

### Smooth rendering techniques

Current S3 performance path:

- Direct display handoff avoids copying every source frame into a display snapshot.
- Triple source framebuffers keep direct handoff safe: one buffer can be displayed while the emulator renders into another.
- Decoded tile-row cache avoids repeated 4-plane tile bit extraction.
- Decoded sprite-row cache avoids repeated 4-plane sprite bit extraction.
- Cached tile-row masks skip rows where every pen is transparent for the current MAME pass.
- Cached sprite-row masks skip rows containing only sprite transparent pen 0.
- Unrolled cached tile-row drawing handles unclipped 8-pixel rows.
- Unrolled cached sprite-row drawing handles unclipped 16-pixel rows.
- Row-level opacity checks avoid per-pixel transparency tests when a cached row has no masked pens.
- Current default cadence: `render_irq=3`.
- Runtime logging suppresses generic ESP-IDF/library chatter and keeps only project diagnostics needed by smoke tests (`rtype_app`, `rtype_rom`, `rtype_core`, and the S3 display tag) at INFO. The device must run from power alone without a serial log consumer.

## Limitations

- No audio output yet.
- No real physical input path yet.
- Not a general MAME/M72 implementation; targets the supplied R-Type path.
- Internal RAM promotion is unsafe on this board because the RGB panel framebuffer needs scarce internal RAM.
- Camera comparison requires a working V4L2 capture device. The tools auto-detect `/dev/video*`, but CI/headless systems may only run serial smoke.
- Dense scenes are still CPU/render-bound; do not trade away complete-frame queue drain for higher nominal IRQ rate.

## Required validation

Minimum firmware validation:

```bash
make build-s3
make smoke-s3 SERIAL_PORT=/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0
```

Full deploy validation:

```bash
make deploy-s3 SERIAL_PORT=/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0
```

If camera is available:

```bash
make compare-s3-host SERIAL_PORT=/dev/serial/by-id/usb-1a86_USB_Serial-if00-port0
```

Do not mark background/fidelity work complete without either:

- a passing host/S3 comparison with visible backgrounds, or
- a passing smoke run with active `qdrain=<nonzero>/0` plus a stated reason why camera comparison is unavailable.
